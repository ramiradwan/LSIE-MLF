Shells

Three layout archetypes power every page. The outer SidebarStackShell wraps the application; pages then compose MetricGridPlusTimelineShell or TableWithDrillDownShell as their inner pattern. Reusing these three is what makes the console feel like one product — a new page does not invent a new layout.

SidebarStackShell Primary application shell · sidebar + stacked pages + persistent ActionBar + status bar overview · live_session · experiments · physiology · health · sessions

The outer chrome of the application. Sidebar holds the fixed six-route nav; the right side stacks pages in a single QStackedWidget that switches via route changes. The ActionBar lives below the stacked content so it stays mounted across routes. Below everything, the status bar reads back the API base URL and environment label so the operator never has to guess which environment they are pointed at.

MetricGridPlusTimelineShell Monitoring shell · summary cards plus a timeline / follow-up region overview · live_session · physiology · health

The console's most-reused inner pattern. A page header sits above a responsive card grid (3/2/1 by width band), with a timeline or detail panel beneath. This shell is where the operator does monitoring — glance at the cards, drop into the timeline only when something asks for attention.

TableWithDrillDownShell Readback shell · table or list adjacent to a detail / management surface live_session · experiments · sessions

When the operator's job is to understand a row, the page splits a table from a detail or management panel. Selection is deliberate: a single click selects, a double-click or Enter opens. The detail panel never auto-loads new content from a hover — the operator owns the navigation gesture.
