Critical 1 — design tokens and widget code disagree

DR-02 "recovering" pill renders in accent blue — StatusPill bypasses the status_recovering token critical
Where it lands
Health · "Getting ready" card. Live Session readiness strip · ml probe. Anywhere the StatusPill widget is asked to show a recovering state.
Root cause
Two contradictions in widgets/status_pill.py:

1. UiStatusKind has no RECOVERING bucket — only OK / INFO / WARN / ERROR / NEUTRAL / PROGRESS / MUTED. So "recovering" must arrive as INFO or PROGRESS, both of which map to PALETTE.accent.

2. An explicit comment in the file says the choice is intentional:
# §12 status colouring. RECOVERING picks the accent colour so it reads # as "self-healing in progress" rather than "warning" (which would # conflate it with DEGRADED).
Why it matters
The design tokens declare a unique hue for this state — status_recovering = #6cc3d5 (teal) — and the QSS uses it for HealthRowRecovering table cells. So the table-cell variant and the pill variant of the same status now render in different colors. The audit (§11.3) names recovering as a distinct color contract; today it's distinct in QSS, accent-collapsed in widgets.
Runtime · StatusPill
recovering
accent #5b8def — same as focus rings and the submit button
QSS · HealthRowRecovering
recovering
status_recovering #6cc3d5 — distinct teal
Decision needed
This is a design-system question, not a bug. Pick one:
  • Option A (recommended) — fix the widget. Add UiStatusKind.RECOVERING, map it to PALETTE.status_recovering, update formatters.py to emit it where appropriate. Restores the audit's color contract.
  • Option B — fix the design system. Drop the status_recovering token and remove HealthRowRecovering; recovering is just accent-with-a-label throughout. Smaller code change but loses a documented UX distinction.
Files
packages/schemas/operator_console/ui_status_kind.py · widgets/status_pill.py · formatters.py (or tokens.py + qss_builder.py if Option B)

Warn 3 — visible runtime drift, single-file fixes

DR-04 Body type falls through to a serif fallback on Linux hosts warn
Where
Everywhere. Sidebar subtitle "Operator Console", ActionBar italic "Expected response: hei rakas", body copy across all six pages.
Spec
qss_builder.py: font-family: "Segoe UI", "Inter", system-ui, sans-serif. Neither Segoe UI nor Inter is installed on the capture host; the fallback chain lands on a serif (likely DejaVu Serif or Liberation Serif).
Why static gates miss it
QSS is valid; the font load is host-dependent. run_audit.py --strict can't tell whether the resolved font is sans or serif.
Fix
Ship Inter as a Qt resource and load it at app boot via QFontDatabase.addApplicationFont(":/fonts/Inter-Regular.ttf") + the bold/italic variants. Guarantees identical rendering on Windows / macOS / Linux. As a stopgap, add DejaVu Sans before sans-serif in the fallback chain so Linux hosts at least land on a sans family.
Files
app.py · design_system/qss_builder.py · services/operator_console/resources/fonts/ (new)
DR-03 "sampling" is an undocumented status word — violates hard-rule #4 warn
Where
Experiments · "Experiment" card → pill labelled sampling in accent blue. Same on Overview · "Experiment" card.
Spec
DESIGN_SYSTEM.md hard-rule #4: "Use UiStatusKind for status mapping rather than ad-hoc string categories." "sampling" is not one of the seven enum values.
Why it matters
"sampling" is a lifecycle phase of an experiment (the strategy is currently selecting arms), not a health or trust state. Painting it identically to recovering/info pills puts it on the same vocabulary tier as the audit-blessed words. Operators now have eight buckets to memorize, not seven.
Fix
Two paths, in order of preference:
  • Demote the phase from a pill to a small mono-12 label next to the experiment ID. Phase is a label, not a status.
  • If it must be a pill, use UiStatusKind.OK with literal copy "sampling" — preserves the color contract while keeping the operator-language word the strategy team prefers.
Files
views/experiments_view.py · views/overview_view.py · formatters.py
DR-05 Live Session: encounter panel overlaps the metric cards above it warn
Where
Live Session at 1024×768 — the panel containing "Why this observed response counted" paints over the bottom of "Ready for a stimulus" and "Live response ticker".
Likely cause
UX-09 introduced a QSplitter(Vertical) with a default 45/55 ratio persisted via QSettings. The captures suggest the splitter's top region is sized smaller than the content's minimum height, and the content's QSizePolicy is Fixed or Preferred rather than MinimumExpanding — so the content extends past the splitter region rather than being clipped or scrolled.
Fix
Set QSizePolicy(MinimumExpanding, MinimumExpanding) on the metric grid container, set a sensible minimumHeight on the splitter's top widget, and consider wrapping the top region in a QScrollArea when its content exceeds the splitter region. Also: persist a sensible minimum splitter handle position so a previous narrow session doesn't trap the layout on relaunch.

views/live_session_view.py

Medium 3 — copy and consistency, single-line fixes

DR-07 "Healthy" (capitalized) breaks the lower-case status word principle medium
Where
Live Session · Session panel reads "Status: Healthy". Every other status word in the console is lower-case (ok, degraded, recovering, idle, fresh, sampling…).
Spec
Operator-language principle #5: "Short, lower-case, no marketing voice."
Fix
Normalise via formatters.py: "Healthy" → "ok" (preferred — matches UiStatusKind) or "healthy" (if the team wants to keep the word). The fact that this slipped through suggests the formatter for session status isn't routing through the same lowercase pipe as the rest.

DR-06 Live Session readiness strip mixes probe vocabulary with action vocabulary medium
Where
Top strip on Live Session.
Observed
Left side carries subsystem probes — phone, ml, audio, video. Right side carries submit, which is an StimulusActionState lifecycle value. Both render with identical StatusPill styling, so the operator must read each label to know which kind of thing it describes.
Fix
Pull "submit" off the readiness strip. Submit lifecycle belongs in the ActionBar (already covered there). If the strip is meant to surface the safe-submit gate result, label it as safe-submit and place it apart from the probes — perhaps a narrow divider then the gate pill, the same way the audit calls out distinguishing categories visually.

DR-08 Health rollup cards duplicate state: primary text is the pill word medium
Where
Health page — "Overall · degraded · degraded", "Needs attention · 1 · degraded", "Getting ready · 1 · recovering". The card's primary text repeats the embedded pill word.
Recommendation
Add a rule in widgets/metric_card.py: skip the embedded pill when primary.lower() already equals the pill's status word. For count-primaries ("1", "0"), keep the pill — count and status convey different things.
