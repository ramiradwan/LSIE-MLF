UX Handoff · Operator Console · LSIE-MLF

UI-only improvements
for engineering handoff

A prioritized set of UI-only recommendations for the current Operator Console, grounded in docs/artifacts/OPERATOR_CONSOLE_UI_UX_AUDIT.md and the existing services/operator_console/design_system/ package. Every recommendation stays inside the existing routes, data, and component vocabulary — no backend, no new pages, no new data collection. Engineering can pick any subset; ordering reflects expected operator-confidence impact per unit of effort.

Strictly in scope

  • Layout, grouping, hierarchy, density
  • Microcopy via formatters.py
  • Existing primitives + QSS in design_system/qss_builder.py
  • Responsive width-band tweaks
  • Accessibility / keyboard / focus

Strictly out of scope

  • New routes, pages, or features
  • API / schema / DB / pipeline changes
  • New data collection or measurements
  • Cloud sign-in, sync, auth surfaces
  • Anything that changes what the system stores

UX-01 Promote SectionHeader sizes to a 3-tier ramp quick win
Surface
Every page header + every panel header.
Problem
Today, page titles and panel titles use almost the same size (15px / 14px). Long sessions confuse which level you're at — page vs. panel vs. card.
Recommendation
Three explicit sizes: SectionHeaderTitle.page = 18/600, .panel = 14/600, .sub = 12/500 muted. Add a level= kwarg to SectionHeader rather than introducing new widgets.
Benefit
Scanability. The operator's eye finds the page heading immediately; panels feel like containers, not competing pages.
No-backend
QSS-only change in qss_builder.py + a single param on the SectionHeader widget.

UX-03 Tabular numerals on every numeric value quick win
Surface
MetricCard primary, table cells with numbers, countdown.
Problem
Proportional digits cause numbers to jitter as values tick — operator's eye keeps re-locking. Especially visible on Live Session (p90, frames, RMSSD) and the countdown.
Recommendation
In QSS, set font-feature-settings: "tnum" 1, "lnum" 1 on MetricCardPrimary, all QTableView cells, ActionBarCountdown, and StatusBarLabel. Apply through qss_builder.py.
Benefit
Numbers stop dancing. Values stay aligned column-wise without manual padding.
No-backend
Pure CSS feature. No widget changes.

UX-04 Visible 2px focus ring on every interactive element medium
Surface
Nav buttons, MetricCard (when clickable), table rows, ActionBar submit, line edits.
Problem
Today only QLineEdit#ActionBarNote:focus changes border. Keyboard navigation is technically present (Live Session auto-selects first encounter, Sessions opens on Enter) but the focused element is invisible. Long sessions on a single keyboard get harder.
Recommendation
Add :focus rules using a 2px outline in --accent, with outline-offset: 2px, on every interactive selector. Pair with QApplication.setStyle's focus-rect setting so Qt actually paints them.
Benefit
Keyboard-only operation becomes practical. WCAG 2.4.7 met. Helps when the operator is mid-stimulus and can't reach the mouse.
No-backend
QSS-only. Behavior already exists; we're just making it visible.

UX-05 Add an explicit muted UiStatusKind for "not configured" medium
Surface
Health probes, Physiology provider, Live Session readiness.
Problem
Today an unconfigured probe and a degraded probe can both end up looking similar (warn-ish). The audit (§13.4) says "not configured is not treated the same as failure" — but the pill ramp doesn't have a slot for it.
Recommendation
Add UiStatusKind.muted with token --text-muted as the dot color. Wire it through StatusPill and HealthRow* selectors. Update formatters.py to emit "not configured" copy.
Benefit
Trust copy matches color. Operators stop second-guessing whether a missing probe is a problem.
No-backend
Existing data already exposes "not configured" states; this only adds a presentation bucket.

UX-06 AlertBanner glyphs as semantic icons, not letters quick win
Surface
AlertBannerInfo / Warning / Critical.
Problem

UX-07 Group cards into 3 named bands: Now · Trust · Needs attention medium
Surface
Overview six-card grid + attention preview.
Problem
Today the six cards (Active Session, Experiment, Physiology, Health, Latest Encounter, Attention) read as a flat grid. The operator does three different jobs on this page — see what's running, judge trust signals, and triage attention — but the layout doesn't reflect those jobs.
Recommendation
Use SectionHeader as group dividers above the grid:
  • Now — Active Session, Experiment
  • Trust — Latest Encounter, Physiology
  • Needs attention — Health, Attention queue
Keep ResponsiveMetricGrid; add a group= arg.
Benefit
Two seconds of scan instead of six card-readings. Mirrors the operator's mental model.
No-backend
Pure layout grouping. No new data.

Live Session

The densest surface — separate operating from inspecting

views/live_session_view.py
UX-09 Split page into two scroll regions: Operate and Inspect later
Surface
All of Live Session.
Problem
Live Session has 12+ regions stacked into a single scroll. During an active stimulus, the operator wants the top half (readiness, telemetry, response signal) glued to the viewport while still being able to scroll the encounter table and detail panel below it.
Recommendation
Use a QSplitter(Vertical) with two regions: top is "Operate" (header pills, readiness, live telemetry, response card, cause/effect), bottom is "Inspect" (encounter table + detail). Persist the splitter handle position to QSettings. Default ratio 45/55. The ActionBar stays shell-mounted as today.
Benefit
Operator can keep the live readback fixed while scrolling history. Cuts scroll-overshoot during measurement windows.
No-backend
Layout-only. All data sources unchanged.

UX-13 Move management panel into a slide-over, not the page body later
Surface
Experiments management panel (create / add arm / rename / disable).
Problem
Mixing readback cards with editable form fields on the same page makes the page feel less authoritative. Edit fields invite "is this safe to type in?" hesitation.
Recommendation
Replace the inline management panel with a "Manage" button in the page header that opens a right-side slide-over (300px wide, modal to the page but not the app). Inside, keep the existing form widgets verbatim. Closes on Esc, save, or click-outside.
Benefit
Page becomes pure readback by default. Edit affordance is deliberate, deliberate-feeling, and reversible.
No-backend
Reuses existing form widgets and signals. No API changes.

UX-16 "null-valid" deserves its own treatment, not a muted MetricCard medium
Surface
Co-modulation panel.
Problem
Today "null-valid" co-modulation is rendered as a MetricCardPrimary in muted gray with the null_reason as secondary. Operators read this as "data missing" — but the spec is clear that null-valid is a real outcome, not an absence.
Recommendation
When the result is null-valid, replace the value cell with a small inline pill (--accent, label "valid · null") and put null_reason in a primary-weight sentence ("Insufficient aligned pairs in window"). Reserve the muted treatment exclusively for actual absence.
Benefit
Operators stop misreading expected outcomes as failures. Reinforces audit principle "informative outcomes, not subsystem failures."
No-backend
Same fields, different presentation.

Health

Diagnostic surfaces deserve diagnostic visualisation

views/health_view.py
UX-17 Probe matrix: row-per-subsystem with status sparkline later
Surface
Health probe matrix + alert timeline.
Problem
Today the probe matrix is point-in-time; the alert timeline is sequence-only. Stitching "is this subsystem flapping?" requires the operator's eye to bounce between regions.
Recommendation
Replace the probe matrix with a row-per-subsystem layout. Each row: subsystem name, current status pill, then a 60-cell horizontal sparkline showing the last 60 probe results (one cell per probe, color = UiStatusKind). Hover a cell shows timestamp + state from the existing alert timeline data.
Benefit
Flapping is preattentive. Recovering vs. degraded vs. error becomes a visible pattern, not a label race.
No-backend
Probe history is already polled and stored client-side. New widget reads existing model.

UX-18 Repair install: confirm dialog + reflect status in the row medium
Surface
Health · Repair Install action and repair status readback.
Problem
Repair install is a heavy action — it reinstalls the runtime — but today it's a normal-looking button. The status readback after the click sits in a separate area.
Recommendation
(a) Wrap the repair button in a QMessageBox confirm with the consequences spelled out. (b) After confirmation, the button morphs to a disabled state showing a determinate progress label ("Installing · step 2 / 4") and the affected subsystem row reflects recovering until completion.
Benefit
Heavy action gets weight. Operator sees the consequence happening on the row that matters, not in a side panel.
No-backend
Repair install already emits status events; we're just binding them to the button.

ActionBar

The only persistent write surface — make every state defensible

widgets/action_bar.py
UX-20 Disable submit visibly, not invisibly: explain why quick win
Surface
ActionBar submit button.
Problem
Today, when safe-submit isn't satisfied, the button silently disables. The operator clicks, nothing happens, and there's no explanation in the bar itself.
Recommendation
When disabled, the ActionBarMessage second-line copy must always state the gating reason from formatters.py:
  • "No session selected — start one to send a stimulus."
  • "Greeting bundle not loaded for arm_b — try repair install."
  • "Measuring active — wait for response window to close."
The disabled button keeps the latest button label (Send Stimulus / Measuring…) instead of going blank.
Benefit
Disabled state stops being mysterious. Operator's "did I click that right?" goes to zero.
No-backend
All gating reasons already exist in the viewmodel; just always render them.

UX-21 Countdown: progress bar under the bar, not just digits medium
Surface
ActionBar countdown during measuring.
Problem
The mono "04:32" countdown is precise but not preattentive. Operators glance at it constantly during the response window.
Recommendation
Add a 2px-tall progress bar pinned to the bottom edge of the ActionBar. Color: --status-warn while measuring; transitions to --status-ok on completed; --status-bad on failed. Width animates from 100% → 0% over the response window. Keep the digit countdown for precision.
Benefit
Time pressure becomes peripheral. Operator can focus on the streamer's reaction without re-reading numbers.
No-backend
Same StimulusUiContext data drives both. New widget, same input.

UX-22 Compact mode (<1024) currently hides note + countdown — collapse instead medium
Surface
ActionBar compact mode.
Problem
Below 1024px the note field and the countdown disappear. The countdown in particular is critical during measuring — losing it on a narrow screen is a regression.
Recommendation
Two-row compact layout instead of hiding:
  • Row 1: session context · pill · submit
  • Row 2 (only when measuring or showing a message): countdown left, message right
Note field collapses behind a small + note chip that expands on click.
Benefit
No critical state ever drops off the bar. Narrow screens stay safe to operate from.
No-backend
Layout-only change.
