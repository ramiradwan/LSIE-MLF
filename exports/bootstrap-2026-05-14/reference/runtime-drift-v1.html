Critical 3 — violate documented audit or vocabulary

DR-01 Cloud sign-in / Cloud outbox UI is present in Health, against audit §14.4 critical
Surface
Health page — top-right toolbar (Cloud sign-in, Update definitions, Repair install) and two bottom cards (Cloud sign-in · Unknown, Cloud outbox · Unknown).
Audit citation
§14.4 and §15: "operator-facing cloud sign-in UI is not currently wired into the operator shell"; the audit's explicit out-of-scope list begins with "operator-facing cloud sign-in UI".
Observed
Two MetricCards stuck on "Unknown · waiting for cloud status / outbox summary" — they never resolve because the producer doesn't exist. A trained operator reads these as broken subsystem, not not wired.
Fix
Either (a) remove the cards and toolbar buttons from views/health_view.py until cloud sync ships, or (b) render the muted UiStatusKind.muted bucket from DR-04 with copy "not configured for this environment" so the operator stops triaging them.

DR-03 "sampling" is an undocumented status word, rendered in accent blue critical
Surface
Experiments · "Experiment" card and Overview · "Experiment" card → pill labelled sampling.
Spec
UiStatusKind in state.py defines six buckets: ok · warn · bad · recovering · degraded · muted. Hard-rule #4 of DESIGN_SYSTEM.md: "Use UiStatusKind for status mapping rather than ad-hoc string categories."
Observed
"sampling" is being used as a free-form status word, painted with the accent. This is exactly the drift the hard rule was written to prevent — every new view that invents a word adds one to the operator's vocabulary load.
Fix
"sampling" describes a lifecycle phase of an experiment, not a health state. Render it as a small StatusPill using UiStatusKind.ok if the strategy is currently selecting actively — or move "phase" into its own non-pill label (mono 11px in muted) so it doesn't compete with the actual health status.
Files
views/experiments_view.py · views/overview_view.py · formatters.py

Warn 4 — visual contract drift

DR-04 Body type is rendering in a serif fallback — Segoe UI / Inter aren't on the host warn
Surface
Every screen.
Spec
qss_builder.py: font-family: "Segoe UI", "Inter", system-ui, sans-serif.
Observed
Sidebar subtitle "Operator Console" reads italic-serif. ActionBar "Expected response" italic uses a serif glyph set. Body copy has noticeable serifs on lowercase d, b, p. This is the QSS font stack falling through to a serif fallback on this host (likely Linux without Segoe UI or Inter installed) — probably DejaVu Serif or Liberation Serif.
Fix
Two options, ranked: (1) ship Inter as a Qt resource and load it via QFontDatabase.addApplicationFont() at app boot — this guarantees the same font across Windows / macOS / Linux; (2) add DejaVu Sans as an explicit late fallback before sans-serif so Linux hosts land on a sans family.
Files
app.py · design_system/qss_builder.py · resources/fonts/Inter-*.ttf (new)
DR-05 Live Session "Why this observed response counted" overlaps the metric cards above it warn
Surface
Live Session · the encounter table panel is painting over the bottom of the "Ready for a stimulus" and "Live response ticker" cards.
Observed
Vertical layout / scroll-area sizing bug. The two metric cards have a fixed-height region; their text overflows under the panel below rather than expanding the row.
Fix
Either give the metric-cards row a QSizePolicy.MinimumExpanding vertical policy, or wrap the content in a QScrollArea. This is the kind of bug the splitter recommendation (UX-09 from the prior handoff) eliminates entirely.

DR-06 Live Session readiness strip mixes probes ("phone", "ml", "audio", "video") with an action label ("submit") warn
Surface
Live Session · top strip.
Observed
Left side reads: phonemlaudiovideo — these are subsystem probes. Right side reads: submit — which is an action lifecycle, not a subsystem.
Why it matters
Two different vocabularies are sitting in the same strip with identical pill styling. The operator has to parse that "phone" and "submit" are different categories of thing.
Fix
Pull "submit" out of the readiness strip. Submit lifecycle belongs in the ActionBar (already covered by StimulusActionState). The strip should only carry readiness probes. If "submit" here is meant as safe-submit gate result, label it as safe-submit and group it visually apart from the other probes.

Medium 4 — layout & consistency

DR-08 Health rollup cards duplicate state: primary value is the status pill word medium
Surface
Health · "Overall · degraded · degraded", "Needs attention · 1 · degraded", "Getting ready · 1 · recovering".
Observed
Card primary text and embedded pill carry the same word. No information added; the card reads as a typo.
Fix
If a card's primary text is a UiStatusKind word, drop the embedded pill. If a card has a count primary ("1", "0"), keep the pill — the two convey different things. Apply this rule once in widgets/metric_card.py: skip the pill when primary.lower() in status_words.

DR-09 StatusPill style differs from spec — outlined dot rather than filled-pill surface medium
Surface
Every page.
Spec
StatusPill background is surface_raised, padded 2/8, dot 8px. The pill is meant to feel like a small filled chip, not a floating dot.
Observed
Pills render as a dot + label on the panel background, no chip. This is fine in isolation but makes scanning a 6-card grid harder because pills don't form an obvious horizontal rhythm.
Fix
Add the background: surface_raised rule for QWidget#StatusPill (it's currently transparent in qss_builder.py — change to surface_raised with border-radius: 999px and padding: 2px 8px). One QSS rule change, no widget code touch.
